## Session start checklist

**Full session (new reports or pages)** — run these three exchanges first:

```
1. Safety
   Confirm: .pbip format, backup taken, Desktop fully closed.
   State clearly that you cannot verify the backup — you are trusting them.
   If they cannot confirm all three, stop and explain what needs to happen first.

   Then ask for the REPORT TITLE — this populates the top-right text box and
   is often missed. Get it now before any visuals are built.

2. Context
   Confirm: audience (Executive / Operational / Mixed), the report
   title (this populates the report title text box in the top bar),
   and that base measures exist in the model before you build visuals.

3. Plan
   Ask for any brief, wireframe, or planning template they have.
   Build a full plan. Show it. Wait for explicit confirmation before touching any file.
```

**Quick session (edits, DAX, documentation)** — acknowledge and proceed, but
always run the end-of-session checklist before finishing.

## How to talk about the layout

When discussing the layout with developers, use plain language. Internal
standards use Zone A/B/C for coordinate precision, but in conversation say:
  Zone A → "the top bar"
  Zone B → "the filter pane"
  Zone C → "the canvas" (the main report area where visuals go)

**Slicer rules:**
- Max **4 slicers** on the top bar. Divide zone x:1599–2350 (751px) equally:

- Slicers must NEVER extend left past x:1599 into the title/subtitle area.
- 5th slicer and beyond → filter pane (Zone B), never on the top bar.
- More Filters button: **HIDE** (visible:false, never delete) when Zone B is
  empty (≤4 slicers total). **SHOW** when Zone B has 1 or more slicers (≥5).

## Colours

The default palette is accessibility-first — blue/orange/grey performs better
for colour-blind users than red/green. Always use this default unless the
developer specifically asks for the red/green alternative.

```
DEFAULT — always use this unless explicitly asked otherwise:
  Good          blue     #017CBE
  Bad           orange   #E3850D
  Neutral       grey     #666666

ALTERNATIVE — only if the developer explicitly requests it:
  Good          green    #02892E
```

## SQL review approach

When reviewing SQL used to import tables into Power BI:

**How to get the SQL:**
- READ IT DIRECTLY from the TMDL partition file in the model
  (e.g. SemanticModel/definition/tables/[table]/partitions/[partition].tmdl)
- The SQL lives inside the M (mashup) expression as a Value.NativeQuery
  or Sql.Database step. Extract it from there.
- Do NOT ask the developer to paste their query — find it yourself.
- If the dev asks "review the SQL for [table name]", read that table's
  partition. If they ask "review all import SQL", read every partition.

**How to review:**
- Explain issues in plain English — no jargon, no technical terms the
  developer did not use first.
- Refer to the data warehouse as "Synapse" (not "the database", not "DW").
- Flag any query that uses complex joins, window functions, running totals,
  or aggregations that Synapse could handle better.
- Recommend building the result as a **table in the reporting schema**
  as part of the **daily batch process**, so Synapse does the heavy work
  once per day rather than at every Power BI refresh.
- Tell the developer how to import the resulting table using named columns —
  never SELECT *.

## Layout patterns — choose the right shape for the purpose

Different pages serve different purposes. Pick the layout that fits the
page's intent — not a blanket default. At planning time, ASK the developer
what each page is for, then choose accordingly.

```
3/30/300        Default for self-service exploration
                3 seconds   what the user grasps at a glance (headline KPIs)
                30 seconds  what they understand after a quick scan (charts)
                300 seconds what they can analyse if they dig in (tables)
                Best when: overview page, mixed audience, no specific story

Narrative       Tell a clear story, walk a stakeholder through findings
                Z-pattern reading flow: top-left → top-right → bottom-left
                → bottom-right. Each visual is one beat of the story.
                Best when: explaining a result, making a case, single
                audience, dev says "tell" / "explain" / "show why"

Executive       Single headline number with supporting context
                One hero visual centred, supporting visuals around it,
                optional detail strip below
                Best when: board reports, exec dashboards, one big number

Operational     Status-focused, no raw data
                Grid of status indicators and small visuals — designed to
                be scanned quickly, no drill-to-row required
                Best when: real-time monitoring, floor-walking dashboards,
                ops reviews
```

**How to choose:**

The agent picks per page based on the developer's stated purpose,
explains the choice, and accepts overrides. For example:

> "Page 1 is the CFO summary — I'll use Narrative since you want to
>  explain the Q4 margin drop. Page 2 is the detail explorer — 3/30/300.
>  Page 3 is the ops board — Operational. Confirm or override?"

Never just announce the layout — always explain WHY it fits the
purpose, so the developer learns the reasoning.

## Authority hierarchy — what wins when things conflict

Two different kinds of "sacred" — content and form. They don't compete
because they're about different things.

```
1. Template STANDARDS — sacred on FORM
   Zones (top bar, filter pane, canvas), coordinates, colours,
   fonts, spacing, slicer widths, legend defaults, vocabulary,
   governance rules
   → NEVER overruled. Template wins on the mechanics.

2. Developer's CONTENT and INTENT — sacred on WHAT
   The visuals they want, the story they're telling,
   the data they want to show, the audience they're serving
   → NEVER overruled. Developer wins on the content.

3. Developer's LAYOUT PREFERENCES — adapted to fit
   If the developer's wireframe puts a visual outside a zone,
   or breaks a spacing rule, or uses a wrong slicer width:
   → agent adapts the placement to fit the template
   → agent explains what changed and why
   → agent gets approval before proceeding

4. Agent's LAYOUT SUGGESTIONS — only when the developer defers
   If the developer says "you decide what fits", the agent picks
   a layout based on the stated purpose and explains the choice.
```

**The rule in plain English:**

```
The developer says WHAT goes on the page.
The template says WHERE it goes and HOW it looks.
The agent brings them together — never bending the template,
never overriding the developer's content.

When they conflict on placement, the template wins.
When they conflict on content, the developer wins.
When they conflict at all, the agent FLAGS it, PROPOSES an
adapted version, and WAITS for approval.
```

**Example of adaptation in practice:**

```
Developer's wireframe:  "5 KPI cards across the top of the page"
Wireframe placement:    y:0–100 (in the top bar / Zone A)
Template rule:          top bar is for slicers and titles only

Agent's response:
  "Your wireframe shows the 5 KPI cards in the top bar area
   (Zone A) — that's reserved for slicers and titles. I've moved
   them to the top of the canvas (Zone C) instead, keeping your
   intent of 5 cards across the top of the page. Coordinates
   would be y:150–260, x:28–2535 divided equally. Sound good?"
```

The agent never silently reshapes. Never refuses. Always adapts
transparently and gets sign-off.

## PBIR JSON syntax rules — critical

When generating PBIR report files, follow these strictly. Mistakes
here cause visuals to fail silently or render as (Blank).

```
projections — ALWAYS an array, even for a single field:
  CORRECT:  "projections": [{ "field": {...} }]
  WRONG:    "projections": { "field": {...} }

filterConfig — also always an array of filter objects
visualContainerObjects — only override when the theme genuinely
  doesn't cover a case. Default behaviour should rely on the theme.
  If you find yourself overriding the same property repeatedly,
  flag that as a THEME GAP for Adam to fix at the theme level —
  do not work around it with per-visual overrides.

TMDL comments — use proper TMDL comment syntax, not // (which is
  C-style and breaks TMDL parsers in some contexts).
```

If you encounter a formatting need that isn't covered by the theme,
do NOT add a workaround. Stop and report: "The theme doesn't appear
to cover [specific case]. This needs to be added to the theme rather
than worked around per-visual. Should I continue with the theme default
or wait for the theme update?"

## Desktop validation checkpoints — render before continuing

Building 16 pages then handing off means errors stack up before they
are caught. Instead, validate at three checkpoints:

```
Checkpoint 1 — after the semantic model phase
  "I have finished the model changes. Please open Power BI Desktop,
   confirm it loads without errors, and tell me what you see before
   I start on the report."

Checkpoint 2 — after the first page is built
  "I have finished page 1. Please open Desktop, confirm it renders
   correctly, and tell me what you see. Catching issues now means we
   do not stack the same mistake across every page."

Checkpoint 3 — before the final end-of-session checklist
  "All pages are built. Please open Desktop and confirm the whole
   report renders before I run the end-of-session checks."
```

At each checkpoint, the agent PAUSES and waits for the developer's
response. Issues reported at a checkpoint are fixed before continuing.
This is non-negotiable for multi-page or complex reports.

```
[ ] No calculated columns added to fact tables
```

Report each as a tick or a flagged issue with the specific page and
visual. Do not declare success while any item is unconfirmed.
